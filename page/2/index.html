Git、SVN

发表于 2017-10-04 更新于 2021-01-17 分类于 Tools

Git与SVN的比较

说是git和svn的比较，倒不如说是分布式版本控制系统与集中式版本控制系统的比较。

去中心化

集中式版本控制，版本库都是存放在中央服务器的，集中式版本控制的问题在于如果我们需要工作，那么必须要连入网络。分布式版本控制系统是没有中心服务器的，每个开发人员的机器上都是一个完整的库，也就是说平时开发代码的中央服务器其实和我们自己机器上的内容是完全一致的（格式会有所不同）。多人协作的场景下，可以相互将修改的文件推送给对方，就可以感知对方的修改了。
在实际的应用中，一般不会这样推送修改，因为不可预测的情况比较多，所以分布式版本控制系统中也有一台充当中央服务器的机器，但是这个服务器的作用仅仅是用来方便交换大家的修改。去中心化意味着没有权威，所有人都是平等的，具体可参见链接。

假如集中式的服务器挂了，那么所有人都挂了，因为完整仓库只存在服务器上，分布式则是如果服务器挂了，那么可以新建一个服务器，然后把任何一个人的仓库clone过去，也就是说分布式版本控制的每个节点都是完整仓库。集中式不能这样做的原因在于除了中央服务器外，其他节点都是没有历史版本的。

本地提交

  1. 断网提交
  2. 小步提交，可以对自己的阶段成果有跟踪
  3. 本地库，和断网提交时同一个实现，也就是说即使只有自己一个人开发项目，也可以轻易的让自己的代码有版本跟踪，而不需要费力建server
  4. 本地回滚，由于本地库的存在，可以在本地进行回滚，但是在提交的时候应该控制，减少中央库的冗余版本

git的本地仓库包含代码库还有历史库，在本地环境开发就可以记录历史。而svn的历史库存在于中央仓库，每次对比与提交代码都必须要连接到中央仓库才行。git将提交版本与推送服务器这两个概念分开了，svn则是融为一体的，提交即推送给服务器。

在git中有几个概念：本地、服务器、中央服务器（远程服务器）。每一次的commit都是提交到本地的服务器，这个不需要联网，这就是所谓的版本管理，我们可以知道每一个版本，回退版本时也可以直接通过本地仓库实现。但是svn得每一次commit都需要联网，非常依赖网络。git只有在push和pull的时候需要联网，建议平时多进行commit。虽然在断网的情况下svn也可以工作，但是在之后的合并阶段会出现较多的问题。

分支策略

分支策略从技术上来讲是将版本节点化了，即最终的版本状态是树状的。从结果上来讲既是弱化了分支，也是强化了分支。

弱化的是分支的概念，强化的是分支的功能。在Git实际开发中分支的分离和merge是属于日常操作，开启和合并分支成本相比SVN要小得多：SVN是复制一份代码到分支目录，Git则是在分支点做一下标记。随便一次冲突就会自动产生分支，所以大家每天都在与分支打交道。这便是弱化了分支的概念，由于分支成本很小，因此使得按功能分支的开发模式（每个分支一个功能，开发完了再merge到主干）变得非常简单，大家可以完全不需要再因为担心SCM成本太高而选用主干开发模式（所有功能都在主干上开发，到了发版本前再分离出分支）。

上述功能作者目前用的比较少，因为涉及到的比较多的是代码提交之类的，反而分支用的不多。

SLF4J和LOG4J

贝叶斯定理

贝叶斯推断是一种统计学方法，用来估计统计量的某种性质。
贝叶斯推断与传统的频率学派不同，它是建立在主观判断的基础上的，也就是说，你可以不需要客观证据，先估计一个值，然后根据实际结果不断修正。由于其主观性太强，所以在一些传统行业内更偏向于传统的频率学派。

贝叶斯定理需要先验知识作为支撑，而先验知识需要大量的计算和历史数据，因此在很长一段时间内，无法得到广泛应用。只有计算机诞生以后，它才获得真正的重视。人们发现，许多统计量是无法进行客观判断的，而互联网时代出现的大型数据集，再加上告诉运算能力，为验证这些统计量提供了方便，也为应用贝叶斯定理创造了条件。

条件概率

$$P( A \mid B ) = \frac {P(A \bigcap B) } {P(B)}$$

$$P(A \bigcap B) = P(A \mid B) * P(B)$$

precision, recall

信息检索、分类、识别、翻译等领域两个最基本指标是准确率（precision rate）和召回率（recall rate），准确率也叫查准率，召回率也叫查全率。这些概念非常重要的一个前提是针对特定的分类器和测试集的，即不同的数据集也会导致不同的结果。

  • True Positives,TP：预测为正样本，实际也为正样本的特征数
  • False Positives,FP：预测为正样本，实际为负样本的特征数（错预测为正样本了，所以叫False）
  • True Negatives,TN：预测为负样本，实际也为负样本的特征数
  • False Negatives,FN：预测为负样本，实际为正样本的特征数（错预测为负样本了，所以叫False）
  • TP+FP+FN+FN：特征总数(样本总数)
  • TP+FN：实际正样本数
  • FP+TN：实际负样本数
  • TP+FP：预测结果为正样本的总数
  • TN+FN：预测结果为负样本的总数

最近的项目有一个需求是需要根据手机传感器的信息，对用户进行动作识别。

动作识别是通过一系列用户动作的记录和环境状况的数据，判断用户的运动状态（motion activity）。项目的需求仅限于根据数据判断用户的运动状态，如步行，跑步，驾车，骑车等。

采集数据采用的手机中的加速度计（accelerometer），Android的核心代码就是记录加速度计的数据以及当前行为状态的真值。采集的频率为 50Hz，综合考虑了耗电和存储的限制。同时在采集数据时，记录了Samsung motion API和Google Activity Recognition的数据，用来和项目最终的效果做一个对比。
